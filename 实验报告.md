## 二、实验要求

### 2.1 项目应用背景

构建一个可以直接应用于校园的二手交易市场，允许同学在线发布二手商品；在后台审核通过之后，可以在线购买下单，锁定商品。同时约定线下交易的时间和地点，通过线下完成整体的交易环节。同时可以针对购买的商品，可以在线进行评论，对于商品和买家进行评论反馈。

该系统由二手交易和后台管理审核系统两个部分组成。后台管理审核，主要是负责对于用户人员、线上发布商品的审核管理功能。二手交易系统主要是提供在线商品的浏览和搜索，并提供用户下单购买的主要流程；同时提供用户注册，提供用户个人信息和购买历史记录等信息。

### 2.2 系统功能

#### 必选功能

1. **用户的注册和登录流程**：注册要求有随机基于图片的验证码。用户信息包括：id、姓名、手机号、邮箱、城市、性别、银行账号（16位数字）。待管理员审核之后，方可正式生效登录使用。

2. **商家注册**：商家注册需要提供营业执照（图片上传）、银行账号、姓名、性别、手机号、身份证（图片上传），审核成功之后，方可正式使用。

3. **角色区分**：区分不同的用户角色（普通用户/商家/管理员）。普通用户：浏览、购买商品、已退货/在途退货商品。商家/普通卖家：发布商品、浏览、下架商品。管理员：审核商品、审核用户注册、管理用户列表（修改/删除/查看）。

4. **商家等级**：商家用户有等级，分为5级，由管理员负责设置。平台根据商家等级收取交易费用。

   | 等级 | 费率 |
   |------|------|
   | Lv.1 | 0.1% |
   | Lv.2 | 0.2% |
   | Lv.3 | 0.5% |
   | Lv.4 | 0.75% |
   | Lv.5 | 1% |

5. **商品发布**：商家在线发布/下架商品（多张图片、价格和数量）。发布商品信息包括：商品名称、类别、价格（原价/折扣）、尺寸大小、照片、使用说明、是否允许议价、商品数量、新旧程度（全新、几乎全新、轻微使用、明显使用）。

6. **商品展示**：支持商家的店铺维度展示商品、商品的库存、历史销量和购买者评价等。

7. **商品评价**：商品支持购买后的评价（基于文字和五星级的评价），评价可以在商品详情页查看。购买用户可以对商家的服务态度进行评价。

8. **中间账户与自动确认**：商家钱包中，在商品售卖之后，超过7天，如买家未确认，则系统自动确认，卖家收到货款。期间此金额放在系统的中间账户中。买家确认收货之后，货款即可打到卖家账户。

9. **购物车与下单**：登录用户可以使用购物车，支持一键下单购物车的商品（后台无需对接支付接口）。购物车中可以暂存多个商家的商品。一键下单时用户账号会被扣除对应的商品货款。

10. **退货机制**：支持用户在收货后24小时内申请退货。在用户发起退货之后，在商家审核之后方可生效；发起退货申请时间，超过收货24小时之后不能退货。

11. **钱包与积分**：用户有钱包功能和积分功能。管理员可以对特定用户进行充值操作。用户购买商品之后，1元消费额即可转换为1个积分。每100个积分可以抵扣1元现金。

12. **商品搜索**：商城需要具备基于商品名称的搜索功能，可以查出多个商家的商品列表，基于价格、销售量和好评率进行排序展示，用户可以设定对应的排序规则。

13. **双向评价**：商家可以对用户的购买/退货商品进行评价，评价会转换为购买者的商家好评率。

14. **发货确认**：商家手动确认发货后，用户方可确认收货。

#### 可选功能

- 商城首页轮播图（不同商家的推荐位以及广告宣传商品）
- 针对不良商家的限时封禁功能和下架功能
- 支持恶意购买者的拉黑功能

### 2.3 技术设计参考

- **数据库表**：User表（普通用户和管理员）、商品表（状态：创建、审核、发布、锁定、交易完成、退货）、订单表、订单明细表、购物车表、退货记录表、评价表、积分日志表
- **核心字段**：用户积分、商品照片/文字描述/价格、用户购买记录、用户评价、商品评价信息

#### 4.1.2 逻辑分工

系统采用经典的四层分层架构：

- **Controller层**：接收HTTP请求，参数校验（@Valid），调用Service，返回统一响应R<T>
- **Service层**：封装业务逻辑，事务管理（@Transactional），乐观锁并发控制
- **Mapper层**：继承MyBatis Plus的BaseMapper，LambdaQueryWrapper构造查询条件
- **Entity层**：Java对象与数据库表映射，@TableName/@TableId/@TableLogic注解

**分层数据流**：
```
HTTP Request → Controller(@Valid校验) → Service(@Transactional事务)
→ Mapper(MyBatis Plus查询) → MySQL → Entity → DTO/VO → R<T> → JSON Response
```

### 4.3 RESTful接口设计规范

#### 设计原则

- URL使用名词复数表示资源，不使用动词
- HTTP方法：GET查询、POST创建、PUT更新、DELETE删除
- 状态码：200成功、400参数错误、401未认证、403无权限、500系统错误
- 统一返回格式：`R<T>{ code: int, msg: string, data: T }`
- 认证方式：Bearer Token（JWT），请求头 Authorization: Bearer xxx
- 角色控制：@RequireRole注解标注接口所需角色

### 4.4 核心模块开发说明

#### 4.4.1 DAO层（数据访问层）

采用MyBatis Plus作为ORM框架，所有Mapper接口继承`BaseMapper<T>`即可获得CRUD能力。配置要点：

- `map-underscore-to-camel-case: true` 实现下划线命名到驼峰命名的自动映射
- `logic-delete-field: deleted` 实现逻辑删除（标记删除而非物理删除）
- 分页插件 MybatisPlusInterceptor 实现物理分页
- LambdaQueryWrapper 编写类型安全的查询条件，避免字段名拼写错误

关键代码：
```java
// 使用LambdaQueryWrapper避免硬编码字段名
List<Product> products = productService.list(new LambdaQueryWrapper<Product>()
    .eq(Product::getStatus, ProductStatus.PUBLISHED.getValue())
    .like(Product::getName, keyword)
    .orderByDesc(Product::getCreatedAt));
```

#### 4.4.2 Service层（业务逻辑层）

核心业务逻辑均在Service层实现，所有写操作添加`@Transactional(rollbackFor = Exception.class)`保证事务一致性。

**订单创建流程**（最复杂的业务逻辑）：
1. 校验购物车归属
2. 计算总金额
3. 积分抵扣计算：`PointsDeductUtil.calcDeduction(points, totalPay)`
4. 钱包扣款：乐观锁 `WHERE wallet >= actualTotal`
5. 积分扣减：乐观锁 `WHERE points >= usedPoints`
6. 库存扣减：乐观锁 `WHERE stock >= quantity`
7. 保存订单和订单明细
8. 赠送积分（消费1元=1积分）
9. 清除购物车

**退货退款流程**：
1. 校验退货期限（24小时内）
2. 恢复库存
3. 退款到买家钱包
4. 从卖家钱包扣回收入（扣除平台费）
5. 扣减买家下单积分

**平台费率计算**：
```java
// 根据商家等级计算平台手续费
public static BigDecimal getPlatformFee(BigDecimal amount, int sellerLevel) {
    return switch (sellerLevel) {
        case 1 -> amount.multiply(new BigDecimal("0.001"));  // 0.1%
        case 2 -> amount.multiply(new BigDecimal("0.002"));  // 0.2%
        case 3 -> amount.multiply(new BigDecimal("0.005"));  // 0.5%
        case 4 -> amount.multiply(new BigDecimal("0.0075")); // 0.75%
        case 5 -> amount.multiply(new BigDecimal("0.01"));   // 1%
        default -> BigDecimal.ZERO;
    };
}
```

#### 4.4.3 Controller层（Web控制层）

**权限校验机制**：自定义`@RequireRole`注解 + `AuthInterceptor`拦截器

```java
// 注解定义
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RequireRole {
    String[] value(); // 允许的角色列表
}

// 拦截器中校验
RequireRole requireRole = handlerMethod.getMethodAnnotation(RequireRole.class);
String role = claims.get("role", String.class);
boolean hasRole = Arrays.asList(requireRole.value()).contains(role);
```

JWT Token中包含userId、phone、role三个信息，拦截器解析后存入ThreadLocal，通过`UserContext.get()`获取当前用户。

#### 代码片段1：IoC/DI的应用 — CartServiceImpl的循环依赖解决

```java
@Service
@RequiredArgsConstructor
public class CartServiceImpl extends ServiceImpl<CartMapper, Cart> implements CartService {
    // @Lazy延迟注入解决与ProductService的循环依赖
    @Lazy
    private final ProductService productService;

    @Override
    public List<CartGroupVO> myCartGrouped() {
        List<Cart> carts = myCart();
        // 使用LinkedHashMap保持分组插入顺序
        Map<Long, CartGroupVO> groupMap = new LinkedHashMap<>();
        for (Cart cart : carts) {
            Product product = productService.getById(cart.getProductId());
            // computeIfAbsent：若分组不存在则创建新分组
            CartGroupVO group = groupMap.computeIfAbsent(product.getSellerId(), k -> {
                CartGroupVO vo = new CartGroupVO();
                vo.setItems(new ArrayList<>());
                return vo;
            });
            group.getItems().add(CartGroupVO.CartItemVO.of(cart, product));
        }
        return new ArrayList<>(groupMap.values());
    }
}
```

#### 代码片段2：复杂事务处理 — 订单创建的乐观锁与积分抵扣

```java
@Override
@Transactional(rollbackFor = Exception.class)
public List<Order> createOrders(OrderCreateDTO dto) {
    // ... 校验购物车、计算金额 ...
    
    // 积分抵扣
    BigDecimal pointsDeduction = BigDecimal.ZERO;
    if (dto.isUsePoints() && user.getPoints() != null && user.getPoints() > 0) {
        pointsDeduction = PointsDeductUtil.calcDeduction(user.getPoints(), totalPay);
    }
    BigDecimal actualTotal = totalPay.subtract(pointsDeduction);

    // 钱包扣款 — 乐观锁
    boolean walletUpdated = userService.update(new LambdaUpdateWrapper<User>()
        .eq(User::getId, userId)
        .ge(User::getWallet, actualTotal)    // 条件：余额 >= 待扣金额
        .set(User::getWallet, user.getWallet().subtract(actualTotal)));
    if (!walletUpdated) throw new BusinessException("扣款失败，余额不足");

    // 库存扣减 — 乐观锁
    boolean stockUpdated = productService.update(new LambdaUpdateWrapper<Product>()
        .eq(Product::getId, cart.getProductId())
        .ge(Product::getStock, cart.getQuantity())  // 条件：库存 >= 购买数量
        .setSql("stock = stock - " + cart.getQuantity()));
    if (!stockUpdated) throw new BusinessException("库存扣减失败，请重试");

    // 积分奖励：消费1元 = 1积分
    int earnedPoints = totalPay.intValue();
    userService.update(new LambdaUpdateWrapper<User>()
        .eq(User::getId, userId).setSql("points = points + " + earnedPoints));

    cartService.removeByIds(dto.getCartIds());  // 清除购物车
    return orders;
}
```

### 方案对比

在处理"中间账户资金托管"时，对比了两种方案：

| 方案 | 描述 | 优点 | 缺点 | 选择 |
|------|------|------|------|------|
| 方案A：独立账户表 | 创建独立的平台账户表，每笔交易资金先转入平台账户 | 账目清晰、审计方便 | 表结构复杂、多一步转账 | ❌ |

最终选择方案B，理由：校园二手交易场景简单、金额小，延迟到账方式足够满足需求。配合7天自动确认的定时任务兜底，确保卖家不会无限期等待。

## 七、实验总结

### 7.1 实验中碰到的问题总结

1. **前后端接口路径不匹配**：前端调用路径与后端Controller的@RequestMapping不一致（如前端`/cart/list`后端`/cart`），通过系统对比所有接口并在vite.config.js添加rewrite规则解决。

2. **字段命名风格不一致**：MyBatis Plus返回camelCase字段名，但前端数据适配器检测的是snake_case字段名，导致detectType失败、adaptProduct不执行。通过更新detectType同时检查两种命名风格解决。

3. **图片上传路径映射**：后端上传路径配置为Linux路径`/data/upload/`，Windows环境需改为`D:/upload/`，且需添加Vite代理转发`/upload`路径。

4. **数据库连接配置**：端口占用、数据库未启动、编码问题等需要逐一排查。

5. **Spring Bean循环依赖**：CartServiceImpl依赖ProductService，而ProductService也可能间接依赖CartService，通过@Lazy注解延迟注入解决。

6. **角色权限值不一致**：后端角色值为中文（"管理员"/"商家"），前端路由守卫初始用英文（"admin"/"merchant"），导致管理员无法访问商家页面。统一改为中文值解决。

7. **业务逻辑缺失**：手动确认收货时缺少打款逻辑、退款时缺少从卖家扣回逻辑、积分抵扣完全缺失，逐一排查并补充。

8. **响应拦截器问题**：后端返回`R<T>`包装，前端需要解包`data`字段。添加响应拦截器自动解包后，多处代码的`res.list`/`res.groups`等访问方式需要同步修改。

### 7.2 Spring Boot使用感想

1. **IoC/DI机制**：Spring的依赖注入大大简化了组件之间的协作。通过`@RequiredArgsConstructor`和`final`字段即可实现构造器注入，无需手动管理对象生命周期。`@Lazy`注解有效解决了循环依赖问题。

2. **自动配置**：Spring Boot的自动配置极大减少了XML配置工作量。MyBatis Plus的自动配置、数据源的自动配置只需在application.yml中设置即可。

3. **事务管理**：`@Transactional`注解使得声明式事务管理非常简单，只需在Service方法上加一个注解即可保证数据库操作的原子性。配合乐观锁的`WHERE`条件，有效防止了并发下的超卖和超扣问题。

4. **拦截器机制**：Spring MVC的拦截器（HandlerInterceptor）使得认证和授权逻辑可以统一处理，不再需要在每个Controller方法中重复编写认证代码。

5. **全局异常处理**：`@RestControllerAdvice`让异常处理变得非常优雅，不同类型的异常可以映射为不同的HTTP状态码，前端可以统一处理。

### 7.3 实验体会

1. **分层架构的价值**：Controller→Service→Mapper的分层让代码职责清晰，修改某一层不会影响其他层。例如修改积分抵扣逻辑只需修改Service层，Controller和Mapper完全不受影响。

2. **前后端分离的效率**：前后端通过RESTful API通信，各自的开发和调试可以独立进行。但联调阶段需要认真对齐接口规范，避免字段名不匹配等问题。

3. **数据适配的重要性**：后端返回的数据格式（snake_case、枚举值）与前端展示所需的格式（camelCase、中文标签）之间需要一层适配层。本次实验的dataAdapter.js较好地解决了这个问题。

4. **乐观锁的实际应用**：在高并发场景下，乐观锁比悲观锁更适合Web应用。通过在UPDATE语句中添加版本条件（WHERE stock >= quantity），避免了锁等待，即使更新失败也可以友好提示用户重试。

5. **团队协作**：本次实验项目的分工明确（一人负责后端+数据库，一人负责前端+联调），通过Git进行版本控制，在开发过程中感受到了真实的软件工程协作模式。

6. **AI辅助编程的认识**：AI编程助手（Claude Code）在本次实验中发挥了重要作用——帮助快速生成模板代码、定位Bug、添加注释、设计测试流程。AI不能替代工程师的系统性思维，但能极大提升编码效率和减少低级错误。
